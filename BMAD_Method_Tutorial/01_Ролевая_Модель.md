# Ролевая Модель в BMAD (Specialized Agents)

В методологии BMAD вы не взаимодействуете с одним «всемогущим» ИИ, который должен делать всё. Вместо этого вы оркестрируете работу **специализированных ИИ-агентов** (или переключаете системные промпты для одной модели). 

Это позволяет ИИ-модели сфокусироваться и применять экспертизу, подходящую для конкретного этапа.

## Виртуальная команда

BMAD выделяет несколько ключевых ролей (Personas), через которые проходит идея:

### 1. Analyst (Бизнес-аналитик)
*   **Задача:** Понять суть проблемы и контекст бизнеса.
*   **Действия:** Проводит мозговой штурм с человеком, задает уточняющие вопросы, выявляет риски и ограничения. 
*   **Результат:** Краткий бриф проекта (Project Brief).

### 2. Product Manager / PM (Продакт-менеджер)
*   **Задача:** Превратить бриф в конкретные требования.
*   **Действия:** Определяет целевую аудиторию, описывает пользовательские сценарии (User Journeys) и ключевые функции.
*   **Результат:** Документ требований продукта (PRD — Product Requirements Document).

### 3. Architect (Системный архитектор)
*   **Задача:** Спроектировать техническое решение на базе PRD.
*   **Действия:** Выбирает технологический стек, проектирует архитектуру баз данных, структуру API, системные компоненты и их взаимодействие.
*   **Результат:** Архитектурный документ (Architecture Design Document).

### 4. Scrum Master / UX Specialist 
*   **Задача:** Разбить большую архитектуру на атомарные задачи.
*   **Действия:** Превращает архитектурный дизайн в набор мелких пользовательских историй (User Stories) с жесткими критериями приемки (Acceptance Criteria).
*   **Результат:** Набор файлов-задач (Step-files или Story-files).

### 5. Developer (Разработчик)
*   **Задача:** Написать код.
*   **Действия:** Получает **одну** конкретную задачу (User Story) и архитектурный документ. Фокусируется только на реализации этой маленькой части, строго следуя спецификации.
*   **Результат:** Готовый код в репозитории.

### 6. QA (Инженер по качеству)
*   **Задача:** Проверить написанный код.
*   **Действия:** Анализирует код на соответствие критериям приемки, пишет юнит-тесты и интеграционные тесты. 
*   **Результат:** Тесты и отчет об ошибках (если они есть).

## Почему это работает?
Когда ИИ играет роль Архитектора, он не думает о синтаксисе конкретной функции — он думает о масштабируемости систем. Когда ИИ играет роль Разработчика, он не думает о бизнес-требованиях — он просто реализует изолированный модуль по готовому чертежу. Такое разделение обязанностей (Separation of Concerns) радикально снижает когнитивную нагрузку на LLM.
