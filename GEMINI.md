# База Знаний AI Dev (Obsidian)

Это не-кодовый проект, хранилище Obsidian. Содержит документацию, лучшие практики и инструменты по AI-разработке.

## 📂 Структура проекта
- **`Новости/`** — Архитектура (Swarms, DAG, DDD), тренды, системный дизайн и управление затратами.
- **`SaveAs/`** — Библиотека ресурсов: LLM-навыки, MCP-серверы, готовые промпты.
- **`BMAD_Method_Tutorial/`** — Описание ролевой модели и фаз разработки.
- **`AI_Project_Startup_Plan/`** — Пошаговый план и инструкции по инициализации и запуску AI-проектов (настройка среды, контракты, роли, TDD).
- **Корень** — Ключевые гайды: `AI_Dev_Workflow_Recommendations.md` (практики ИИ) и `AI_Pixel_Perfect_Design.md` (верстка по контрактам).

---

## 🤖 Инструкции для Агента (Workflow Rules)

### 1. "Импортируй новости телеги"
* **Действие:** Проанализируй переданный текст или ссылки.
* **Локация:** Строго распределяй информацию по существующим тематическим папкам и файлам внутри директории `Новости/`. Запрещено создавать единые сводные файлы-свалки (дампы) для новостей. Если тема принципиально новая — только тогда создай новый `.md` файл в подходящей подпапке.
* **Формат:** Сделай выжимку сути (TL;DR), структурируй в чистый Markdown, добавь релевантные теги для Obsidian.

### 2. "Синхронизируй знания из других инструкций"
* **Действие:** Изучи предоставленные инструкции (например, из других проектов, `CLAUDE.md`, `docs/`).
* **Интеграция:** Глубоко встрой новые правила в существующие документы без простого копирования в конец файла:
  - UI/UX, вёрстка, работа с Figma → `AI_Pixel_Perfect_Design.md`
  - Общие AI-подходы, архитектура агентов → `AI_Dev_Workflow_Recommendations.md`
  - Запуск проектов, инициализация, базовые настройки → `AI_Project_Startup_Plan/`
  - Новые MCP-серверы / навыки → `SaveAs/`
* **Очистка:** Удаляй устаревшие или противоречащие новому подходу правила. Не плоди дубликаты.

### 3. Общие конвенции
* **Приоритет поиска:** При вопросах об инструментах или архитектуре сначала ищи ответы локально в `.md` файлах проекта.
* **Связи Obsidian:** Сохраняй работоспособность Markdown-ссылок (вида `[[Название файла]]` или относительных).
* **Git Workflow:** После любых изменений в базе знаний, предлагай запушить результат в удаленный репозиторий.
