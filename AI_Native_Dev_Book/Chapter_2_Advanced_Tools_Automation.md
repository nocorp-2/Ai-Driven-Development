# Глава 2. Продвинутые инструменты и автоматизация (Advanced Tools & Automation)

Современная AI-разработка (AI-Native Dev) давно ушла от формата «общения в чате в браузере». Сегодня разработчик выступает в роли архитектора, управляющего полноценными агентными пайплайнами прямо в терминале (с использованием Claude Code, Codex и других CLI-инструментов). В этой главе мы разберем продвинутые концепции, которые позволяют масштабировать работу искусственного интеллекта на огромные кодовые базы и сложные задачи без потери контекста и качества.

---

## 1. Subagents и Свармы (Multi-agent v2)

Монолитный ИИ-агент начинает «буксовать» или забывать детали на масштабных задачах, затрагивающих десятки файлов. Это связано с так называемым «дефицитом внимания» (attention deficit) у LLM — при перегрузке контекстом даже топовые модели упускают краевые кейсы. Решением этой проблемы стали **Свармы (Рои) и Субагенты**.

### Иерархия и делегирование
Сложные эпики разбиваются на изолированные куски и делегируются специализированным ИИ-ролям (персонам). В современных пайплайнах применяется жесткая иерархия:
- **Master Agent (Оркестратор):** Координатор верхнего уровня. Помнит бизнес-цели, долгосрочный контекст, управляет расписанием и не пишет код напрямую.
- **Specialized Sub-agents:** Узконаправленные исполнители (Frontend Developer, Backend, Blog Writer, QA). Каждый имеет свой профиль и специфичные инструкции (например, через реестр `AGENTS.md`).

### Agent Mail (Агентная почта)
Если раньше субагенты запускались просто как дочерние сессии, то сегодня применяется асинхронная координация (Agent Mail). Главный агент создает задачу в виде Markdown-контракта (например, `task-123.md`) и отправляет его субагенту. Субагент (или пул субагентов) выполняет работу, записывает результат в другой файл и «пробуждает» менеджера ответом. Это позволяет главному оркестратору сохранять контекстное окно чистым.

### Мультисэмплинг и "Совет ботов" (Council)
- **Мультисэмплинг (Plan Swarming):** Инструмент для поиска скрытых архитектурных багов. Оркестратор запускает 5–25 агентов-ревьюеров на *один и тот же план* параллельно. Это сжигает много токенов, но «компьют заливает недостаток внимания», окупаясь отсутствием багов в продакшене.
- **Совет ботов:** Привлечение альтернативных моделей (например, GPT-4o, Gemini, Claude) в качестве независимых экспертов для выхода из архитектурных тупиков или при зацикливании основного агента на одной ошибке.

## 2. MCP (Model Context Protocol)

Для того чтобы ИИ-агент мог читать файлы, ходить в интернет, выполнять bash-команды или делать SQL-запросы, ему нужны «руки и глаза». Эту роль выполняет стандартизированный протокол **MCP (Model Context Protocol)**.

MCP позволяет декларативно подключать внешние инструменты (интеграции) к агенту, обычно через файл конфигурации `mcp-servers.json`.

### Ключевые принципы работы с MCP:
1. **Must-have интеграции:** 
   - **Context7** (или аналоги): автоматическое получение актуальной документации по библиотекам и фреймворкам. Предотвращает галлюцинации и использование ИИ устаревших API.
   - **Sequential Thinking:** инструмент, заставляющий агента выстраивать строгое пошаговое дерево рассуждений при дебаге.
   - **Playwright (или chrome-devtools-mcp):** дает агенту «глаза» для открытия браузера, инспекции верстки, сравнения со скриншотами из Figma и проверки консоли.
   - **Git / GitHub:** для автономного чтения PR, работы с ветками и создания коммитов.

2. **Оптимизация токенов и защита памяти:**
   - **Проблема:** Каждый MCP-сервер передает агенту системные описания своих инструментов, сжигая контекст (токены) и засоряя память. Возможны утечки памяти при избытке дублирующихся MCP.
   - **Решение:** Жестко ограничивайте количество серверов (не более 10) и доступных инструментов (до 80). Отключайте (или убирайте из конфига) неиспользуемые серверы для текущего этапа (например, отключайте Git MCP на этапе сбора бизнес-требований).

3. **Безопасность (AgentShield):**
   Никогда не коммитьте `mcp-servers.json` с реальными токенами. Используйте загрузку через переменные окружения ОС или `.env`.

## 3. Hooks (Хуки и гейты качества)

ИИ склонен писать код, который «выглядит правильно», но ломается при компиляции. **Хуки (Hooks)** — это автоматизированные скрипты жизненного цикла (обычно в папке `.claude/hooks/`), которые устанавливают железную инженерную дисциплину. 

Агент не может завершить задачу, пока не пройдет все установленные гейты.

### Типы Хуков:
- **`PreToolUse` (Гейты безопасности):** Скрипты-блокировщики, проверяющие команду до её выполнения. Они могут заблокировать опасные команды вроде `rm -rf`, `git push --force` или предотвратить отправку секретов в логи ИИ.
- **`PostToolUse` (Линтеры и Тесты):** Запускаются сразу после того, как агент редактирует файл.
  - Настраивается автоформатирование (Prettier, Ruff).
  - Запускаются линтеры и тайп-чекеры (например, `eslint .`, `tsc --noEmit`, `mypy .`).
  - Прогоняются unit-тесты (`jest`, `pytest`). Если тесты падают, агент получает лог ошибки и обязан исправить код, прежде чем идти дальше.
- **Хуки контекста:** Сохранение, суммаризация и сжатие контекста перед закрытием сессии, чтобы при следующем запуске агент загрузил бриф и продолжил работу (Continuous Learning).

## 4. Практические кейсы

### Написание тестов для legacy-кода
**Сценарий:** У вас есть старый, недокументированный проект.
**Решение:** Используется навык **Research-first**. Оркестратор спавнит несколько дешевых субагентов, которые параллельно читают старый код и строят карту зависимостей. Затем применяется жесткий пайплайн (TDD): ИИ пишет падающие тесты на существующие функции, прогоняет их (срабатывают хуки тестирования) и затем рефакторит код, добиваясь зеленого света от тестов. 

### Документирование больших кодовых баз
**Сценарий:** Нужно создать `Architecture.md` для монолита.
**Решение:** Вместо того чтобы заставлять одного агента читать все файлы и переполнять контекст, используется подход **Progressive Disclosure** (древовидное раскрытие). Десяток субагентов параллельно анализируют разные директории и пишут локальные сводки в `.tasks/`. Верхнеуровневый агент собирает из этих сводок итоговую, стройную документацию.

### Дебаг сложных багов
**Сценарий:** Плавающая ошибка, логи непонятны, агент топчется на месте.
**Решение:** Агент переводится в режим `Sequential Thinking`. Он обязан явно записать в файл список гипотез. Затем он использует Playwright MCP для воспроизведения шагов в браузере и сбора логов. Если агент делает более 3 неудачных попыток, вызывается команда `/council` — сторонние модели (GPT-4 / Gemini) анализируют тупик и предлагают альтернативный вектор расследования.

### AI-assisted Code Review
**Сценарий:** Субагент или джуниор написал большую фичу.
**Решение:** Запускается независимая сессия ИИ с переключенной ролью «Строгий Senior Reviewer». Включается **Plan Swarming**: 5–10 агентов параллельно инспектируют Diff. Они проверяют код на соответствие архитектурным контрактам (из `CLAUDE.md`), чекают производительность и уязвимости. Только после того, как все замечания устранены, срабатывает хук `PreToolUse` для `git commit`, разрешая слияние кода.
