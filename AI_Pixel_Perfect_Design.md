# 🎨 Pixel-Perfect верстка: Полное руководство

Руководство по достижению пиксель-в-пиксель соответствия дизайну, используя правильный флоу, контракты, MCP и верификацию.

---

## 📋 Принцип: Планирование > Кодирование

Как и в AI Dev: **правильный контракт и спеки дают лучший результат, чем быстрое кодирование**.
Не спешишь прямо в CSS — сначала зафиксируй контракт с дизайном и убедись, что все инструменты работают.

### 🤖 AI Behavior Rules (Karpathy-style)
- **Think Before Coding** — явно проговаривай предположения перед изменением.
- **Simplicity First** — минимальный код, без преждевременных абстракций.
- **Surgical Changes** — меняй только то, что нужно по задаче. Соседние файлы не трогать.
- **Single Source of Truth** — `tokens.scss`. Хардкод цветов/отступов в компонентах **запрещён**.
- **No Improvisation on Design** — нет ноды Figma → не верстаем «по аналогии».
- **Stop & Ask** — в зонах риска (см. ниже) сначала план, потом узкий диф, при сомнениях стоп.

**Risk Zones (требуют согласования с пользователем):**
- Токены (`tokens.scss`, `variables.scss`) — изменения только после аппрува.
- Зафиксированные секции из `bd memory`.
- Конфигурационные файлы (например, `nuxt.config.ts`, `package.json`, `tsconfig.json`).
- Любая destructive операция (`rm -rf`, `git reset --hard` и т.д.).

## 🛑 Фаза 0: Подготовка и Stop-conditions

Без выполнения этих условий приступать к вёрстке **строго запрещено**:

1. **MCP Figma:** В `.mcp.json` подключены инструменты Figma MCP (`mcp__figma-desktop__get_design_context`, `mcp__figma-desktop__get_screenshot`, `mcp__figma-desktop__get_variable_defs`).
2. **Перезапуск:** ИИ-агент (Claude Code) перезапущен после правки конфигурации.
3. **Проверка связи:** Вызов `mcp__figma-desktop__get_metadata` отдаёт ноды файла без ошибок.

> **Золотое правило:** Визуальное совпадение с макетом ≥ 95% на 1440 / 768 / 390. Любое отклонение — только по согласованию.

### Регистрация новых макетов (Обязательно)
Когда появляется **новый** Figma-макет / node-id — фиксируем его **до начала вёрстки**:
1. **`docs/pages-map.md`** — для страниц (добавить запись: «Страница / Figma node-id (1440/768/390) / состав секций → компоненты»).
2. **`docs/blocks-inventory.md`** — для блоков/компонентов (добавить в нужный уровень: Foundation / Atoms / Molecules / Organisms / Layouts / Pages с node-id).

*Node-id всегда в формате markdown-ссылки, указываются все брейкпоинты.*

## 🎯 Фаза 1: Токены и Setup

### 1.1 Atomic Flow
Вёрстка идёт строго снизу-вверх: **tokens → atoms (UI-кит) → molecules → organisms → templates (layouts) → pages.**
Не верстать страницу, пока все её блоки не собраны и не приняты.

### 1.2 Design Tokens (CSS Custom Properties)
Создай переменные ДО кодирования стилей (обычно в `tokens.scss`):

```css
:root {
  /* Colors */
  --color-primary: #2E7D32;
  --color-bg-primary: #F5F5F5;
  /* Typography, Spacing (8px grid), Border radius, Shadows... */
}
```
**Хардкод магических чисел и hex в компонентах запрещён.** Всё через `tokens.scss`.

## 🎨 Фаза 2: Pixel-Perfect Workflow (DAG Вёрстки блока)

Для каждого блока соблюдается строгий DAG и контракт ([`docs/design-spec.md`](docs/design-spec.md)):

```
Figma node → get_design_context + get_screenshot
        ↓
Извлечь измерения (отступы, размеры, цвета, типографика, тени, радиусы)
        ↓
Свериться с tokens.scss; если токена нет — добавить и согласовать с пользователем
        ↓
Сверстать BEM + SCSS scoped (или CSS Modules), ссылки только на токены
        ↓
chrome-devtools-mcp: открыть на 1440 / 768 / 390, сравнить со скриншотом Figma
        ↓
Запустить post-block-checklist (docs/post-block-checklist.md)
        ↓
Зафиксировать «магические» решения в bd remember
```

### Размеры и Сетка (8px подход)
- Все отступы = multiples of 8px (8, 16, 24, 32...).
- Font-size, line-height и color должны быть **ровно как в макете**. Не использовать расчеты или полупрозрачность вместо точных HEX-цветов.

## ✅ Фаза 3: Верификация

### 3.1 Screenshot Comparison (Figma vs DevTools MCP)
Сравнивай свою вёрстку с экспортом из Figma (`mcp__figma-desktop__get_screenshot`). 
**Автоматизированная проверка:**
Используй инструменты браузерной автоматизации (например, `chrome-devtools-mcp`), чтобы:
1. Открыть сверстанную страницу или компонент на целевых брейкпоинтах (1440 / 768 / 390).
2. Сделать скриншот реализованного UI (viewport screenshot).
3. Провести визуальное overlay-сравнение скриншота из Figma со скриншотом из браузера для подтверждения Pixel-Perfect совпадения.

### 3.2 Checklist верификации
- [ ] Container width и padding корректны для брейкпоинта (Desktop, Tablet, Mobile).
- [ ] Все элементы лежат на 8px сетке.
- [ ] Typography (размер, вес, line-height) точь-в-точь по спеку.
- [ ] Цвета (HEX) соответствуют токенам.
- [ ] Отсутствуют хардкод-значения (всё на CSS Variables).

## ⚙️ Фаза 4: Трекинг и Интеграция с Beads (bd)

Если проект использует **bd (beads)** для трекинга задач, соблюдайте **жесткие правила**:

- Используйте `bd` для ВСЕХ задач — **НЕ ИСПОЛЬЗУЙТЕ** TODO-листы в Markdown, TodoWrite или TaskCreate.
- Команды: `bd ready` (поиск работы), `bd show <id>`, `bd update <id> --claim`, `bd close <id>`.
- Для персистентных знаний используйте `bd remember` — **НЕ ИСПОЛЬЗУЙТЕ** `MEMORY.md` файлы.

### MANDATORY WORKFLOW (Завершение сессии)

Когда заканчиваешь рабочую сессию, **ОБЯЗАТЕЛЬНО** выполни все шаги. Работа не закончена, пока push не успешен:

1. **Создай issues для оставшейся работы** — если что-то нужно доделать, добавь в трекер.
2. **Запусти quality gates** — тесты, линтеры, сборка (если код менялся).
3. **Обнови статусы в bd** — закрой завершенное, обнови in-progress.
4. **PUSH TO REMOTE (КРИТИЧЕСКИ ВАЖНО)**:
   ```bash
   git pull --rebase
   bd dolt push
   git push
   git status  # MUST show "up to date with origin"
   ```
5. **Очисти локальные данные** — убери stashes, удали ненужные ветки.
6. **Проверь** — все изменения должны быть закоммичены и отправлены (pushed).
7. **Передай контекст** — оставь summary для следующей сессии (hand off).

**Критические правила завершения:**
- НИКОГДА не останавливайся до успешного push (иначе работа застрянет локально).
- Если push не удался, исправь ошибку и повтори.
