# Шаг 5: Spec-Driven Development (Планирование и Спецификации)

**Статус:** 🔴 Обязательно

## Зачем это нужно?
Самая частая ошибка при работе с ИИ — попросить его сразу "написать код для фичи X". Это приводит к ошибкам, переписыванию и трате токенов. Spec-Driven Development означает **жесткую блокировку написания кода до завершения этапа проектирования**.

## Гейты (Этапы проектирования)

### 1. Research (Исследование кодовой базы)
Прежде чем что-то менять, агент обязан изучить структуру проекта и существующий код.
*Инструмент:* Команда или хук `/search-first` (например, из экосистемы `everything-claude-code`).

*Пример использования в Claude Code:*
> "/search-first Найди все места, где используется аутентификация, чтобы мы могли добавить новый OAuth-провайдер."
*(Агент не будет писать код, он только просканирует файлы и выдаст отчет).*

### 2. Spec (Создание спецификации)
Формирование четких технических требований на основе исследования.

*Пример команды:*
> "На основе нашего ресёрча, напиши техническую спецификацию (Spec) для внедрения GitHub OAuth. Сохрани в файл `auth-spec.md`."

### 3. Plan (Формирование архитектурного плана)
Построение пошагового плана реализации спецификации (blueprints).

*Пример команды:*
> "/plan Создай blueprint для реализации `auth-spec.md`. Распиши по шагам, какие файлы нужно изменить, какие создать, и какие зависимости установить."

**Итог:** И только после того, как вы (человек) прочитаете и утвердите сгенерированный план, агенту разрешается приступать к написанию кода!
