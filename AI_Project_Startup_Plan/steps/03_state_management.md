# Шаг 3: Настройка системы управления контекстом и задачами (State Management)

**Статус:** 🔴 Обязательно

## Зачем это нужно?
Модели LLM (в т.ч. Claude) со временем забывают первоначальную задачу (эффект "снежного кома контекста") или начинают "галлюцинировать", если история чата слишком большая. Управление состоянием (task-tracking) решает эту проблему, заставляя агента вести дневник в репозитории.

## Memory Bank (через bd remember)

**Memory Bank** — это база знаний, в которой агент хранит актуальное состояние проекта, контекст и важные решения. Ранее для этого использовались отдельные markdown-файлы, однако сейчас мы используем команду `bd remember` для более надежного управления памятью.

Агент должен фиксировать ключевые решения, контекст и прогресс напрямую в базу памяти, избегая засорения проекта множеством текстовых файлов.

### Инструкция по использованию:
1. **Сохранение памяти (вручную или агентом):**
   Когда нужно запомнить важное архитектурное решение или обновить текущий статус, используется команда:
   ```bash
   bd remember "Мы решили использовать PostgreSQL вместо MySQL для нового микросервиса из-за поддержки специфичных JSONB индексов. Текущая задача: настройка подключения."
   ```

2. **Миграция из старых файлов:**
   Если в проекте уже существуют старые файлы памяти (например, `projectbrief.md`, `activeContext.md` в папке `memory-bank/`), вы можете перенести их в базу. Для этого попросите агента:
   > "Прочитай файлы в директории `memory-bank/`, сохрани все ключевые факты и контекст через `bd remember`, а затем удали эти файлы."

## Борьба с Context Rot (Протуханием контекста)
Длительные сессии неизбежно ведут к деградации качества ответов. Необходимо активно управлять контекстом путем ручной компрессии до того, как сработает автоматика.

**"Правило двух коррекций":**
Если агент застрял на ошибке, и два ваших объяснения/исправления не помогли решить проблему — не продолжайте спорить. Очистите контекст (начните новую сессию `/compact` или перезапустите) или вызовите "Совет ботов" для независимого взгляда.

## Альтернатива: Beads и интеграция с Dolt

**Beads** — это фреймворк, использующий похожий подход: хранение стейта в графе Markdown файлов. Отлично подходит для сложных проектов с большим количеством взаимосвязей. Отличительной особенностью Beads является интеграция с **Dolt** (SQL-базой данных с версионированием, как в Git) для надежного хранения и синхронизации состояния проекта.

### Инструкция по настройке Beads + Dolt
*(Рекомендуемый рабочий процесс)*

1. **Установка Dolt:**
   Предварительно убедитесь, что в вашей системе установлен [Dolt](https://github.com/dolthub/dolt).
2. **Инициализация Beads:**
   Выполните в корне вашего проекта команду:
   ```bash
   bd init
   ```
   > ⚠️ **Важное предупреждение (Git Flow):** Инициализация `bd` (и шаблоны из хуков/CLAUDE.md) по умолчанию может требовать от агента выполнения коммита и пуша (`git push`) в конце сессии. Если вы находитесь в ветке `master` и у вас нет строгих правил защиты веток (branch protection), это может привести к прямым пушам в `master` без PR. Всегда проверяйте, что вы находитесь в правильной рабочей ветке (например, `develop` или feature-ветке), прежде чем делегировать агенту пуш в конце сессии.

   В результате создастся скрытая директория. Перейдите в неё:
   ```bash
   cd .beads/embeddeddolt/<имя_вашего_проекта>
   ```
3. **Настройка пользователя:**
   Задайте глобальные параметры email и имени (используйте ваши корпоративные данные):
   ```bash
   dolt config --global --add user.email "ivanov.ivan@zoloto585.ru"
   dolt config --global --add user.name "Иванов Иван"
   ```
4. **Подготовка базы и ключа (Web UI):**
   - Зайдите в веб-интерфейс вашего сервера Dolt.
   - Создайте новую базу данных.
   - Сгенерируйте и скопируйте свой персональный ключ доступа.
5. **Применение ключа доступа:**
   В терминале (в папке `embeddeddolt`) выполните:
   ```bash
   dolt creds use <ваш_ключик>
   ```
6. **Подключение удаленного репозитория (Remote):**
   Свяжите локальную базу с удаленной. *Пример:*
   ```bash
   dolt remote add origin http://10.165.0.151:50051/admin/zoloto585-frontend
   ```
   *(Замените `admin/zoloto585-frontend` на формат `<владелец>/<имя_базы>`)*
7. **Отправка данных (Push):**
   Если ваша удаленная база еще пустая, отправьте в неё данные:
   ```bash
   dolt push origin main
   ```
   *(Если в базе уже есть данные, этот шаг можно пропустить)*
8. **Возврат в рабочую область:**
   Выйдите из служебной папки обратно в корень проекта:
   ```bash
   cd ../../../
   ```
9. **Ежедневное использование:**
   В дальнейшем все команды для работы с состоянием выполняются через обертку `bd`. Например, для того чтобы стянуть последние обновления из базы (если она уже была создана ранее):
   ```bash
   bd dolt pull
   ```
