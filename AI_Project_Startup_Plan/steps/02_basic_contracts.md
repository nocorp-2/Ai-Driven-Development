# Шаг 2: Создание базовых контрактов проекта (Единый источник правды)

**Статус:** 🔴 Обязательно

## Зачем это нужно?
ИИ-агентам нужны жесткие правила игры и контекст. Если каждый раз объяснять агенту, какой используется стек или стиль форматирования, вы быстро исчерпаете лимит контекста и получите спагетти-код. Контракты фиксируют эти правила "в камне".

## Файловая структура и примеры

### 1. `CLAUDE.md` / `GEMINI.md` (Глобальные правила и Локальные контексты)
**Современный подход (Отказ от гигантских складов документации):**
Вместо одного гигантского файла `CLAUDE.md` на весь проект, который быстро устаревает и "съедает" токены, используется иерархический подход:
- **Глобальный `CLAUDE.md` (в корне):** Содержит только карту проекта (project map), зоны риска (high-risk zones, где агент должен остановиться и спросить), глобальные инварианты, команды для запуска гейтов качества (линтеры, тесты) и базовые AI Behavior Rules. (Около 100 строк).
- **Локальные `CLAUDE.md` (в каждом модуле):** Стек конкретного модуля, его инварианты, команды сборки, нюансы. Агент получает эту информацию только когда заходит в соответствующую директорию.

*Пример структуры для глобального `CLAUDE.md`:*
```markdown
# Project: My SaaS App
Stack: React, TypeScript, TailwindCSS, Node.js (Express).

## Mandatory Commands (Обязательные шаги после изменения кода)
- Lint: `npm run lint`
- Test: `npm test`

## Risk Zones (Зоны риска)
- Изменение схемы БД (файлы в `db/migrations/`).
- Настройки авторизации и безопасности (`src/auth/`).
> При работе в этих зонах: сначала план, узкий диф, при сомнениях СТОП и эскалация на пользователя. Вне этих зон действуй автономно.

## AI Behavior Rules (Andrej Karpathy Style)
- **Think Before Coding:** Явно формулируй предположения.
- **Simplicity First:** Пиши минималистичный код.
- **Surgical Changes:** Меняй ТОЛЬКО то, что необходимо.
- **Single Source of Truth:** Единственный источник правды — это КОД и контракты данных. Не дублируй логику в текстовых документах.
```

### 1.1. Контракты обмена данными (Zod / OpenAPI)
Вместо текстовых описаний форматов данных, создаются строгие контракты (например, схемы Zod), которые лежат в одном месте (например, `packages/shared/src/schemas/`). Backend является владельцем этих схем, а Frontend и Mobile их импортируют. Это устраняет расхождения и ловится на этапе тайпчекинга (Guardrails).

> **Инсайт:** Секция *AI Behavior Rules* вдохновлена [правилами Андрея Карпати (Andrej Karpathy)](https://github.com/forrestchang/andrej-karpathy-skills/blob/main/CLAUDE.md). Они помогают бороться с главной проблемой LLM — склонностью к "галлюцинациям предположений" и оверинжинирингу. Модель заставляют "замедлиться", проговорить план и делать строго хирургические, минималистичные правки (без переписывания соседних функций, которые агент "не понял").

### 2. Директория `rules/` (Специфичные правила с фильтрами)
Чтобы избежать «засорения» памяти модели и не раздувать `CLAUDE.md`, складывайте детальные гайды в папку `.claude/rules/`. 

**Glob-фильтры (Frontmatter):**
Используйте frontmatter в Markdown файлах правил, чтобы они применялись автоматически только для определенных файлов. Это реализует паттерн иерархии контекста.

*Пример файла `.claude/rules/typescript.md`:*
```markdown
---
globs: ["**/*.ts", "**/*.tsx"]
---
# Правила TypeScript
Используй строгую типизацию. Избегай `any`.
```

### 3. `SKILL.md` (Навыки агента)
Формализация конкретного навыка (например, деплой на AWS или рефакторинг UI). Используется для прогрессивного раскрытия: агент сначала читает список доступных навыков, а затем подгружает нужный `SKILL.md` только при необходимости.

*Пример использования:*
Создайте `skills/aws-deploy/SKILL.md`, где пошагово описано, как деплоить ваш проект. В `CLAUDE.md` укажите агенту: "Для деплоя изучи навык aws-deploy".

### 4. `AGENTS.md` и `DESIGN.md`
*   **`AGENTS.md`** — выполняет роль "Желтых страниц" (адресной книги) для вашего проекта. Описывает, какие субагенты (роли) доступны в проекте, их зоны ответственности и как с ними взаимодействовать.
    *Зачем вписывать туда стандартных агентов из `everything-claude-code`?* 
    Это критически важно для оркестрации. Чтобы главный агент (Генералист) сам решал, кому делегировать подзадачу (через Agent Mail), ему нужен справочник доступных "сотрудников" (Планировщик, Архитектор, Ревьюер). Без этого файла главный агент может попытаться сделать всё сам.
    
    *Пример `AGENTS.md`:*
    ```markdown
    # Реестр ИИ-Агентов проекта (Swarms)
    
    Главный агент должен делегировать специфичные задачи этим ролям:
    
    ## 1. Planner (Планировщик)
    - **Цель:** Декомпозиция крупных фич, создание эпиков и пошаговых планов.
    - **Вход:** Описание бизнес-требований.
    - **Выход:** Файл `plan.md` с пошаговыми чек-листами.
    
    ## 2. Architect (Архитектор)
    - **Цель:** Проектирование структур БД, API контрактов.
    - **Вход:** `plan.md`, `CLAUDE.md`.
    - **Выход:** Файлы в `docs/architecture/`.
    ```

*   **`DESIGN.md`** / `design-spec.md` — строгие UI/UX контракты и дизайн-токены.
    *Пример команды агенту:* "Перед версткой компонента кнопки, сверься с `DESIGN.md` для получения актуальных цветов и отступов."
