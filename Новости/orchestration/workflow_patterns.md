# Паттерны рабочих процессов: Markdown как API

Главный сдвиг в парадигме ИИ-разработки — **отказ от общения с LLM в формате диалога в чате**.

## Контрактное взаимодействие через `.md` файлы
Разработчики осознали, что долгие сессии с LLM приводят к "снежному кому контекста" и быстрой потере внимания у модели.

**Как решают проблему:**
1. **Материализованные планы:** Вся постановка задач, планирование и результаты (отчеты субагентов) фиксируются в виде `Markdown`-файлов.
2. **Рабочие папки (например, `.tasks/`):** Для каждой задачи создается своя директория. В ней лежат черновики, спеки, артефакты исследований субагентов.
3. **Отслеживаемость (Traceability):** Всегда можно открыть файл и посмотреть оригинальное задание. Агент лучше ставит развернутую задачу в файл, чем в общий контекст чата. 
4. **Рефлексия:** Используется паттерн рефлексии. Сначала в файл пишется черновик плана. Затем на него натравливается промпт: "доработай задачу, проверь на наличие пробелов (gaps)".
5. **Memory Banks:** Иерархическое хранение знаний. Агентам запрещено напрямую "срать" в исходники памяти, для этого существуют строгие гейты (через MCP) с проверкой на паттерны (например, progressive disclosure).

## Непрерывность и автономность (Case: Wiz)
Эволюция от "инструмента по запросу" к "автономному сотруднику" (Personal AI Agent):
- **Continuity (Непрерывность):** Агент больше не живет в рамках одной сессии. Он сохраняет состояние между запусками, помня предпочтения пользователя и статус задач в Notion/Linear.
- **Two-level Memory (Двухуровневая память):**
    - *Short-Term:* Оперативный контекст (~50-100 строк). Профиль пользователя, текущие цели, резюме последних действий.
    - *Long-Term:* Проиндексированные Markdown-файлы или RAG, подгружаемые только по запросу.
- **Auto-Wake (Автоматический запуск):** Использование системных планировщиков (типа `launchd` или `cron`) для пробуждения агента по расписанию (например, подготовка утреннего отчета к 7:00).

## Стейт машины и "ИИ-Сотрудники"
Соло-разработчики создают сетапы, неотличимые от конвейеров в компаниях, где есть роли тестировщика, ревьюера, архитектора, но все они выполняются ИИ.

## Методологии гибкой разработки с ИИ
Одной из передовых методологий является **BMAD-METHOD** (Breakthrough Method for Agile Ai Driven Development). Это комплексный фреймворк, где ИИ выступает не просто как исполнитель, а как эксперт-коллаборатор, ведущий пользователя через структурированные воркфлоу (анализ, архитектура, реализация).

**Особенности BMAD-METHOD:**
1. **Scale-Domain-Adaptive:** Автоматическая подстройка глубины планирования под сложность проекта.
2. **12+ специализированных агентов:** Разделение ролей на PM, Архитектора, Разработчика, UX-дизайнера и др.
3. **Party Mode:** Режим совместного обсуждения задачи несколькими ИИ-персонами в одной сессии.
4. **Test Architect (TEA):** Модуль для стратегии тестирования на основе рисков.
