# Переход на AI-разработку с Claude Code, Пайплайны и DDD

**Кратко:** 
Обсуждение глобального сдвига в подходе к написанию кода. Переход от «общения в чате» с ИИ к полноценным агентным пайплайнам (Claude Code, MCP), где разработчик выступает в роли архитектора. Управление «свармами» (субагентами), внедрение систем трекинга задач для ИИ прямо в репозиторий (Beads) и использование принципов Domain-Driven Design (DDD).

---

### 1. Введение: Сдвиг парадигмы в ИИ-разработке
* **Отказ от классического чата:** Долгие сессии с LLM — это тупик. Они приводят к «снежному кому контекста» и дефициту внимания у модели. В больших задачах ИИ начинает забывать детали и галлюцинировать. [Подробнее о паттернах рабочих процессов](workflow_patterns.md)
* **Концепция «Лошадь и Кучер» (LLM + Harness):**
  * Сама LLM («Лошадь») — это лишь двигатель.
  * Успех кроется в «Упряжке» (Harness/Framework) — инструментах контроля и инфраструктуре. Реверс-инжиниринг Claude Code показал, что 98.4% кодовой базы агента — это инфраструктура (тулы, парсинг, безопасность), и лишь <2% — сама логика LLM. [Архитектура и системный дизайн](system_design.md)

### 2. Инструментарий: Claude Code и расширения (MCP)
* **Claude Code как база:** Перенос ИИ прямо в терминал для работы с кодовой базой напрямую. Для удобства можно рассмотреть UI-обертки вроде `cmux` (терминал с табами, работающий как Cursor для CLI). [Инструменты и фреймворки](tools_and_frameworks.md)
* **Единый источник правды:** Внедрение обязательных файлов `CLAUDE.md`, `AGENTS.md` (и `DESIGN.md` для фронта) в репозитории для фиксации стандартов, стека и правил проекта. [Инструменты и фреймворки](tools_and_frameworks.md), [AI инструменты дизайна](ai_design_tools.md)
* **Полезные MCP Skills (навыки для Claude Code), которые нужно внедрить:**
  * Интеграция с базой знаний (например, Obsidian через *Claude-mem*) в качестве «второго мозга» агента.
  * Специфичные навыки: *Architecture Designer*, *Microservice Architect*, *API Designer* для грамотного проектирования до написания кода. *(источник: папка SaveAs/llms_and_skills)*

### 3. Task-трекинг и инфраструктура для ИИ (State & Workflows)
* **Проблема потери контекста:** ИИ-агентам нужна своя система управления задачами, которая живет прямо в репозитории и понятна моделям. Мы больше не держим задачи только в внешних системах. [Инструменты и фреймворки](tools_and_frameworks.md)
* **Инструменты-аналоги (Harness-экосистема для агентов):**
  * **Beads:** Полноценный task-tracking для ИИ. Отвечает за состояние проекта (state). Хранит задачи прямо в `git`, сохраняет контекст между сессиями, учитывает зависимости и блокировки задач.
  * **Superpowers:** Слой инженерной дисциплины. Управляет процессом разработки, заставляя агента жестко проходить фазы: мозговой штурм (brainstorming), планирование, TDD и верификация результата.
  * **Template Bridge:** Связующее звено, которое раскладывает идеи на эпики и подзадачи, встраивая их в рабочий процесс агентов. 

### 4. Пайплайны для LLM и архитектура агентов
* **Свармы (Swarms) и Субагенты (Multi-agent v2):** Разделение сложных задач на изолированные куски. Вместо того чтобы один агент делал всё, задача делегируется пулу субагентов, общающихся через «Агентную почту» (Agent Mail). [Свармы и субагенты](swarms_and_subagents.md)
* **Мультисэмплинг (Plan Swarming):** Генерация нескольких вариантов решения проблемы параллельно для покрытия всех краевых кейсов.
* **Интеграция с CI/CD:** Автоматизация рутины в существующих пайплайнах (GitLab CI). Агенты должны не только писать код, но и покрывать его автотестами (особенно для микросервисов и старого легаси), а также участвовать в раскладке на разные платформы. *(источник: папка Roadmap/Проекты)*

### 5. Разработка с нуля: DDD и Spec-Driven Development
* **«Новый мировой порядок»:** Мы больше не просим ИИ «просто написать код». Мы составляем грамотное ТЗ для написания кода с нуля, опираясь на предметно-ориентированное проектирование (Domain-Driven Design).
* **Принудительное планирование (Spec-Driven-Development):** Жесткая блокировка написания кода до завершения подготовительных этапов:
  1. *Исследование (Research)*
  2. *Спецификация (Spec)*
  3. *План и архитектура (Plan)*
  4. *Написание тестов (Tests)*
  5. *Написание кода (Code)*

### 6. Сбор инсайтов и открытое обсуждение (Q&A)
* **Обмен опытом:** Кто в команде уже использует ИИ в работе? Какие инструменты (Cursor, Copilot, ChatGPT, Claude)? Что болит, что получается хорошо?
* **Роль разработчика:** Обсуждение того, как меняется роль от «кодера» к системному архитектору («Кучеру»), который отвечает за DDD, проверку ТЗ, настройку ИИ-пайплайнов (Beads/Superpowers) и код-ревью.
* **Следующие шаги:** Выбор пилотного проекта (или микросервиса) для обкатки нового пайплайна с Claude Code, DDD и файлами-задачами в git. Формирование базового файла `CLAUDE.md`.
