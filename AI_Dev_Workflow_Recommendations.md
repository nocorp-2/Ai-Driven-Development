# Как разрабатывать свои флоу в AI Dev?

Сейчас все кто работает в AI Dev уже волей-неволей наработали свой флоу, свой подход к решению задач. 

Кто то юзает стандартные упряжки, кто то тестирует и пользует разные инструменты, кто то сделал свои.
Кто то берет набор фреймворков и юзает, кто то пилит свои. 

## Ключевые вопросы:

### 1️⃣ Базовая упряжка? В какой операционной среде работаете?
### 2️⃣ Флоу? Какими процессами пользуетесь?
### 3️⃣ Тулы? Какой то дополнительный AI SWE tools?

---

## 💡 Практические подходы с опыта community

### ▶️ **Подход #1: Memory Bank + Custom Orchestrator**

**Базовая упряжка:**
- Claude Code + GPT модели + Copilot + OpenCode (с разными, в основном "китовыми" моделями)

**Флоу:**
- Основан на **Memory Bank** и сформированных подходах
- Активное изучение фреймворков (superpowers, beads, openspec, speckit, bmad) с адаптацией
- Использование **swarm** в Code для параллельного сбора информации и кодирования
- Отработанные рутины → в **custom orchestrator** для детерминированного выполнения
- Экспериментальные потоки → ручной режим
- **Lessons learned** → кристаллизуются в подход, закрываются "дыры" во флоу

**Tools:**
- Stitch (для UI)
- cmux (оркестратор или удаленная работа)
- Custom orchestrator (рутина)
- Agent browser, Playwright, Agentation (специфические задачи)
- **Skills** (операционные ранбуки, девопс-автоматизация, мониторинг)

---

### ▶️ **Подход #2: Жесткая фиксация контрактов (Anti-spaghetti)**

**Вместо длинных флоу:**
- **Логические гейты** между этапами
- Жесткая фиксация **контрактов, спецификаций, goals, non-scope**
- **Context engineering** вместо сложных флоу
- Результат: избегаются "спагетти-флоу" и путаница

---

### ▶️ **Подход #3: DAG-ориентированная оркестрация**

**Структура:**
- Классический **DAG** (направленный ациклический граф)
- **Не циклящиеся**, четко направленные процессы
- Промпт агентам: "DAG" → не циклится, направлено
- Эпик → задачи → подзадачи (строгая последовательность)

**Верификация:**
- Несколько уровней проверки
- Разные агенты + разные модели
- Четкие критерии качества

---

### ▶️ **Подход #4: Долгое планирование (ACFS / Agent Flywheel)**

**Система от Jeffrey Emanuel:**
- **Agent Flywheel System (ACFS)** — полная система с памятью и оркестрацией
- Memory банк + оркестрация + работа с разными LLM провайдерами
- **Долгое планирование** экономит токены на переделку (back propagation for agents)

**Практика:**
- Brief → План (разными моделями: Sonnet для плана, Haiku для кода)
- Feature.new → feature.plan → feature.implement → feature.promote-to-epic
- Сохранение артефактов: brief, план, саммари
- Документирование в репо

---

## ⚡ Антипаттерны и что НЕ работает

❌ **Недоинструктированные агенты** → творят странные вещи  
❌ **Длинные сложные флоу** → становятся спагетти в коде  
❌ **Неправильный план** → плохой результат (неважно как кодишь)  
❌ **Много параллельных субагентов без координации** → конфликты и хаос  
❌ **Запуск агентов на произвол судьбы** → блуждают хаотично  
❌ **Только текстовые команды оркестратором** → недостаточно  

**Решение:** Задавай через файлы, используй шаблоны, верифицируй артефакты.

## 🎯 Best Practices из обсуждения

1. **Планирование > Кодирование**  
   Инвестируй в правильный план — это сэкономит токены и время

2. **DAG вместо сложных флоу**  
   Направленные, не циклящиеся процессы = понятнее для агентов

3. **Memory Bank = экономия токенов**  
   Накапливай знания, lessons learned, лучше воспроизводи успехи

4. **Один агент = одна задача**  
   Проще управлять и верифицировать чем параллельные рои

5. **Разные модели для разных этапов**  
   Sonnet/Opus для планирования, Haiku для реализации

6. **Автоматизируй рутину**  
   Отработанные процессы → детерминированные скрипты/skills

7. **Lessons Learned → Адаптация**  
   Не просто исправляй баги, найди "дыру" во флоу где они возникают

8. **Контракты и гейты вместо свободных флоу**  
   Жесткие спеки, цели, non-scope = меньше хаоса

9. **Multi-layer verification**  
   Разные агенты, разные модели, четкие критерии

10. **Используй проверенные фреймворки**  
    superpowers, beads, speckit, ACFS — адаптируй под свою задачу

## 🔄 Цикл развития флоу

```
Эксперимент (ручной) 
    ↓
Lessons Learned (анализ)
    ↓
Адаптация флоу (добавление гейтов/спеков)
    ↓
Автоматизация (в оркестратор/skill)
    ↓
Масштабирование (применение к похожим задачам)
    ↓
Memory Bank (документирование)
    ↓
(Новый эксперимент)
```

---

**Главный takeaway:** Правильный план, четкие контракты и долгая адаптация дают результаты лучше, чем сложные автоматизации без понимания процесса.
